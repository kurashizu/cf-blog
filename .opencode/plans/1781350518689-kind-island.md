# Plan: D1 Schema Design for Blog + News Index

## Overview

Move all content indexes from R2 JSON files to D1. Blog article `.md` content stays in R2; only the listing/metadata goes to D1.

### Storage split

| Data | Before | After |
|---|---|---|
| Blog article `.md` files | R2 `articles/*.md` | R2 (unchanged) |
| Blog index (listing) | R2 `cache/articles-index.json` | **D1 `posts` table** |
| HN news (latest 5) | R2 `cache/hn-news.json` | **D1 `news_items` + query LIMIT 5** |
| HN news archive | R2 `cache/hn-news-index.json` | **D1 `news_items` table** |
| Other caches (github, contributions, llm) | R2 | R2 (unchanged) |

## 1. D1 Schema (`d1/schema.sql`)

```sql
-- Blog articles index
CREATE TABLE posts (
    slug        TEXT PRIMARY KEY,
    title       TEXT NOT NULL,
    date        TEXT NOT NULL,          -- ISO 8601
    description TEXT DEFAULT '',
    tags        TEXT DEFAULT '[]',      -- JSON array string
    published   INTEGER NOT NULL DEFAULT 1,
    draft       INTEGER NOT NULL DEFAULT 0,
    cover_image TEXT DEFAULT '',
    author      TEXT NOT NULL DEFAULT 'kurashizu',
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_posts_date ON posts(date DESC);
CREATE INDEX idx_posts_published ON posts(published) WHERE published = 1;

-- HN news archive (with AI summaries)
CREATE TABLE news_items (
    id          INTEGER PRIMARY KEY,   -- HN story id
    title       TEXT NOT NULL,
    url         TEXT,
    score       INTEGER NOT NULL DEFAULT 0,
    by          TEXT NOT NULL DEFAULT 'unknown',
    time        INTEGER NOT NULL,      -- unix timestamp
    descendants INTEGER NOT NULL DEFAULT 0,
    domain      TEXT,
    summary     TEXT NOT NULL DEFAULT '',  -- LLM-generated MD summary
    fetched_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_news_time ON news_items(time DESC);

-- Track which days have been fetched (prevents duplicates on 30-min cron)
CREATE TABLE news_fetch_log (
    date        TEXT PRIMARY KEY,       -- '2026-06-13'
    count       INTEGER NOT NULL DEFAULT 0,
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
```

## 5. Cache-Worker Changes

### Blog index step (`articles-index`)

Currently: `buildArticleIndex()` → writes JSON to R2

After: `buildArticleIndex()` → writes to D1 `posts` table via `INSERT OR REPLACE` for each post. Also keeps R2 write for backward compatibility (or we can remove it).

The query:
```sql
INSERT OR REPLACE INTO posts (slug, title, date, description, tags, published, draft, cover_image, author)
VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?)
```

### HN news step (`hn-news`)

After fetching + summarizing 5 stories:

1. Check `news_fetch_log` for today's date
2. If not fetched today: `INSERT OR IGNORE` stories into `news_items`, log date
3. No R2 writes needed for news

### Page queries (replacing R2 reads)

**Blog listing** (`app/blog/page.tsx`):
```sql
SELECT slug, title, date, description, tags FROM posts
WHERE published = 1 AND draft = 0
ORDER BY date DESC
LIMIT 20 OFFSET ?
```

**News archive** (`app/news/page.tsx`):
```sql
SELECT * FROM news_items
ORDER BY time DESC
LIMIT 20 OFFSET ?
```

**Homepage news (latest 5)**:
```sql
SELECT * FROM news_items
ORDER BY time DESC
LIMIT 5
```

## 6. Admin Editor: D1 Write on Save

### Current flow:
```
PostEditor → POST/PUT /admin/api/posts → repo.save(slug, content) → r2Put to R2
```

### After D1:
```
PostEditor → POST/PUT /admin/api/posts → repo.save(slug, content)
  → r2Put to R2 (unchanged — article .md stays in R2)
  → parse frontmatter from content, INSERT OR REPLACE into D1 posts table (NEW)
```

### `lib/articles.ts` — modify `save()`

```typescript
import matter from "gray-matter";
import { getDB } from "@/lib/d1";

async save(slug: string, content: string): Promise<void> {
    const key = r2Paths.article(slug);
    await r2Put(key, content);

    // Sync to D1
    try {
        const parsed = matter(content);
        const meta = parsed.data;
        const db = getDB();
        await db.prepare(`
            INSERT OR REPLACE INTO posts
            (slug, title, date, description, tags, published, draft, cover_image, author)
            VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?)
        `).bind(
            slug,
            meta.title || "",
            meta.date || new Date().toISOString().split("T")[0],
            meta.description || "",
            JSON.stringify(meta.tags || []),
            meta.published ? 1 : 0,
            meta.draft ? 1 : 0,
            meta.cover_image || meta.coverImage || "",
            meta.author || "Kurashizu",
        ).run();
    } catch (e) {
        console.error("D1 sync failed:", e);
    }

    // Invalidate old R2 cache index
    try {
        const { r2Delete } = await import("@/lib/r2");
        await r2Delete(r2Paths.articlesIndexCache);
    } catch { /* ignore */ }
}
```

This keeps the `save()` method self-contained. The D1 write is best-effort (try/catch) so a D1 failure never blocks the article save.

## 8. Migration Steps

```bash
# 1. Create D1 database
npx wrangler d1 create cf-blog-db

# 2. Apply schema
npx wrangler d1 execute cf-blog-db --file=d1/schema.sql

# 3. Backfill existing data
# Blog posts: read from R2 articles/*.md, parse frontmatter, insert into D1
# HN news: read from existing R2 cache/hn-news-index.json, insert into D1
```

A one-time backfill script will be written at `scripts/backfill-d1.ts` (or inline as a `wrangler d1 execute` command).

## Performance

D1 queries with `ORDER BY ... DESC LIMIT 20 OFFSET ?` on indexed columns execute in **single-digit milliseconds** regardless of row count. No full-file fetch + parse + slice.
