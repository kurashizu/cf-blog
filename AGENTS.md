## D1 Database

| Table | Purpose |
|---|---|
| `posts` | Blog articles — frontmatter fields (slug, title, excerpt, cover_image, tags, author, status, published_at) **plus the full body** in the `content` column (frontmatter stripped). Also tracks `content_hash` and `search_updated_at` for the search pipeline. |
| `news_items` | HN news archive (id, title, url, score, by, time, descendants, domain, summary, fetched_at, search_updated_at) — `fetched_at` is set by the daily cron |
| `github_repos` | Full list of public GitHub repos with per-repo language breakdown (top 3 langs with %) |
| `cache_entries` | Generic key-value cache (replaces R2 for small caches). Stores LLM leaderboard and GitHub contributions. |

Both `cf-blog` and `cf-blog-cache` have `DB` binding to the same D1. Cache-worker syncs GitHub repos, LLM leaderboard, and contributions into D1; cf-blog reads from D1 directly. Blog content is the source of truth in D1 — written directly by the admin API, no R2 mirror.

Schema reference: `database/schema.sql`

## Site-Wide Search (Vectorize + Gemini Embedding 2)

| Component | Location | Purpose |
|---|---|---|
| Vectorize index | `blog-search` (768 dims, cosine) | Stores chunk embeddings |
| 3-min cron | `cache-worker/src/handlers/search-index.ts` | Index one dirty item (blog or news) per tick |
| 30-min cron | `cache-worker/src/lib/refresh.ts` | Refreshes GitHub repos, LLM leaderboard, contributions caches |
| Search API | `cf-blog/app/api/search/route.ts` | `GET /api/search?q=...` — embeds query → Vectorize query |
| Search page | `cf-blog/app/search/page.tsx` | SSR search results page |
| Search bar | `cf-blog/components/search/SearchBar.tsx` | Header search icon/input |

### How it works

1. **Blog content change detection** — Admin API (`cf-blog/lib/articles.ts` `save()`) computes `SHA-256(title + excerpt + first 500 chars of body)` and writes it as `content_hash` on every save. Uses `ON CONFLICT` to set `search_updated_at = NULL` when content changes (first save or hash differs), marking it dirty for the search indexer.
2. **News summary → dirty** — Heartbeat sets `search_updated_at = NULL` on the news item after generating its summary.
3. **3-min `handleSearchIndexing()`** — Each tick:
   - Finds a dirty blog post (blog priority) → reads `content` from D1 → `chunkItem()` → `embedBatch()` → `upsert()` → marks clean
   - If no dirty blog, finds a dirty news item → same pipeline
   - If nothing dirty → idle (no cleanup needed — orphan vectors are harmless and rarely occur)
4. **Search API** — Embeds query with `"task: search result | query: {q}"` prefix, queries Vectorize with `topK=15`, returns matches with metadata (source, title, heading, excerpt).

### Chunking strategy

Blog and news content both go through `chunkItem()` in `cache-worker/src/lib/chunker.ts`:
- **Overview** chunk (blog only): title + description + tags
- **Section** chunks: split by `##` headings
- **Paragraph** chunks: if a section exceeds 1200 chars, split by `###` → paragraphs → sentences
- Each chunk is formatted: `"title: {title} | text: {content}"` (Gemini Embedding 2 document format)

### Vector metadata schema

```typescript
{
  source: "blog" | "news";
  type: "overview" | "section" | "paragraph";
  id: string;          // blog slug or "news-{id}"
  title: string;
  heading: string;      // section heading (empty string for overview/paragraph)
  excerpt: string;      // ~150 char preview
  tags: string[];       // blog tags
  url: string;          // news original URL
  by: string;           // news author
  published_at: string; // ISO date
}
```

```bash

```

## Cache Worker

`cache-worker/` is a cron-triggered Cloudflare Worker that pre-fetches all external data into D1. The homepage never calls GitHub / Artificial Analysis on user requests.

| Cache key | Storage | Source | Notes |
|---|---|---|---|
| D1 `github_repos` | D1 | GitHub REST API | All public repos (paginated), fetches per-repo language breakdown (`/repos/*/languages`) for top-3 % display |
| D1 `posts` | D1 | Admin editor | Body stored in `content` (frontmatter stripped). cf-blog reads from D1 only |
| D1 `news_items` | D1 | Daily cron (top 30) | Daily 05:00 UTC cron pulls top 30 from HN; `INSERT OR REPLACE` keeps the table dedup'd. Summaries generated by heartbeat. cf-blog reads from D1 only |
| D1 `cache_entries:github-contributions` | D1 | GitHub GraphQL | Needs `GITHUB_PERSONAL_ACCESS_TOKEN` secret |
| D1 `cache_entries:llm-leaderboard` | D1 | Artificial Analysis API | Needs `ARTIFICIAL_ANALYSIS_API_KEY` secret |

Schema reference: `database/schema.sql`

Manual trigger:
```bash
curl -X POST https://cf-blog-cache.kurashizu123.workers.dev/__refresh \
  -H "Authorization: Bearer $CRON_SECRET"
curl -X POST https://cf-blog-cache.kurashizu123.workers.dev/__heartbeat \
  -H "Authorization: Bearer $CRON_SECRET"
```
Response is `{ success, logs }` with one `OK/FAILED/SKIPPED` line per cache. Useful for diagnosing cron failures without `wrangler tail`.

### Heartbeat (3-min cron)

A separate `*/3 * * * *` cron processes pending news item rewrites one at a time. Each tick:
1. Queries D1 for a `news_items` row with empty `summary`
2. Fetches the article URL, strips HTML (and extracts image URLs)
3. Calls `gemma-4-31b-it` via Gemini API to produce a full-length Markdown rewrite
4. Response parts with `thought: true` are filtered out; only non-thought content is stored
5. Updates the row's `summary` field

The 30-min cron no longer calls Gemini or HN — it only refreshes caches. The `__heartbeat` endpoint also triggers this manually.

### Daily HN fetch (cron)

A `0 5 * * *` cron calls `handleHNCron()` which:
1. Calls `fetchHNNews(30)` to pull HN's top 30 stories
2. `INSERT OR REPLACE` into `news_items` (the same id on subsequent days just overwrites)
3. `fetched_at` is auto-stamped via the column DEFAULT `(datetime('now'))`

The homepage reads the top 5 by HN time: `ORDER BY time DESC LIMIT 5`. The `/news` archive shows the full history.

## Agent Worker (cf-agent)

Separate Next.js Cloudflare Worker using the same `@opennextjs/cloudflare` build as `cf-blog`. URL: https://agent.022025.xyz. No R2 — talks to Gemini directly for chat + tool calling. Default model: `gemma-4-31b-it`.

### Build & Deploy

```bash
cd agent-worker
npm ci --legacy-peer-deps    # package-lock.json required
npm run dev                   # Local dev
npm run build:cf              # Build for Cloudflare Workers
npx wrangler deploy           # Direct deploy (skips CI)
```

Bindings in `wrangler.toml`: `GEMINI_MODELS` env var, KV namespace (1-hour session TTL), rate limiters.

### Tool System

All tools live under `lib/tools/` and are **manually imported** — `fs.readdirSync` does NOT work in Cloudflare Workers.

**To add a new tool:**
1. Create `lib/tools/<tool-name>.ts` implementing the `Tool` interface
2. Import and register in `lib/tools/index.ts` `TOOL_LIST`
3. No other changes — `/api/tool` and `/api/chat` auto-discover via the registry

```typescript
interface Tool {
    name: string;            // "eval_expression"
    description: string;     // Human-readable, sent to Gemini
    example: string;         // Example usage JSON
    parameters: object;      // JSON Schema → functionDeclaration
    execute(args: Record<string, unknown>): Promise<unknown>;
}
```

| Tool | Description | Parameters |
|---|---|---|
| `eval_expression` | Safe JS expression evaluator (recursive-descent, no `eval`/`Function`) | `{ code: string }` |
| `web_search` | Brave Search API | `{ q: string }` |
| `get_time` | Current time in timezone (UTC fallback) | `{ timezone?: string }` |
| `blog_search` | Blog semantic search (Vectorize + Gemini Embedding) | `{ q: string, topK?: number, source?: "all" | "blog" | "news" }` |
| `blog_read` | Read full article Markdown by slug | `{ slug: string, maxLength?: number }` |

### API Endpoints

- `GET /api/tool` — list all tools with names, descriptions, examples, and Gemini `functionDeclarations`
- `POST /api/tool` — execute a tool: `{ name, args }` → `{ success, result, tool }`
- `POST /api/chat` — streaming chat with tool-calling loop

**SSE event order** for `/api/chat`:
```
start_process
  ↓ (optional tool iterations, max 5)
tool_start → tool_result → end_tool
  ↓
start_text → text → end_text
  ↓
end_process
```

Think blocks (`<think>...</think>`) are filtered before text is emitted. Session messages stored in KV with 1-hour TTL.

### Key Files

```

  lib/
    tools/
```

## Hero Data Widgets (cf-blog)

The homepage hero has two data widgets, both transparent (no card chrome):

1. **Contributions ribbon** — single horizontal row of 51 cells (14×14 each, day-by-day), shows the user's last ~2 months of GitHub activity. Month labels (Apr/May/Jun) above the cells at month boundaries.
2. **Top languages donut** — 100×100 SVG donut in the hero right column, with compact vertical legend. Hover arcs or legend items to see the language name + percentage in the center. Color-coded using GitHub's official language palette.

Contributions powered by `cf-blog-cache` cron → R2, rendered server-side via `getContributions()`. Languages read from D1 `github_repos` table (all non-fork repos, not just top 6) via `getTopLanguages()` from `lib/languages.ts`.

## Conventions

- All Next.js workers use `@opennextjs/cloudflare` (not `next start`).
- Static assets go in `public/`. Blog article markdown lives in D1, not the repo (R2 markdown backups were removed in the D1 migration).
- Theme colors via CSS variables in `components/theme/tokens.css`. Tailwind utilities map to these (`bg-bg-card`, `text-text-muted`, etc.).
- CSS is split by scope: `components/theme/tokens.css` (variables) + `components/theme/base.css` (reset + body effects + terminal/logo + `prefers-reduced-motion`) are global in `app/layout.tsx`. Page-specific CSS is co-located with components (`components/activity/activity.css`, `components/nes/nes.css`) and imported only where needed. All other styling uses Tailwind utility classes directly in JSX.
- Component files: one per file, filename matches component name. `"use client"` only when needed.
- For R2 operations (upload API only), use the Cloudflare REST API or `wrangler r2 object` (the Cloudflare MCP API has known issues, and current wrangler versions don't ship a `list` subcommand — use a Worker with `env.PUBLIC_BUCKET.list()` to enumerate, or the REST API).
- Delete obsolete code completely — no commented-out blocks, no `_unused` files, no half-removed components. If a file is dead, `git rm` it.

### Animation System

- **Page transition**: `PageTransition` in layout (`components/layout/PageTransition.tsx`) handles route-change fade-in. On navigation, content hides instantly (`opacity: 0`, no transition) then fades in via double `requestAnimationFrame`. No exit animation — App Router doesn't support it reliably.
- **Element entrance**: All staggered entrance classes (`animate-fade-up`, `animate-fade-up-sm`, `animate-fade-zoom`, `animate-fadeIn`, `animate-scaleIn`, `animate-slideUp`, `animate-slideDown`) use `ease-out` at 0.18–0.3s. No spring/scale.
- **VisitorTerminal cursor**: Static accent-colored block, no blink. Typewriter effect at 16ms/char.
